src/graph/
Đây là toàn bộ code của module B1 — Graph RAG & Knowledge Base.
schema.py — định nghĩa Cypher schema cho Neo4j/FalkorDB: node types (MenuItem, Chunk, Entity), relationships (NEXT, MENTIONS, BELONGS_TO). Chạy một lần khi init DB, idempotent.
client.py — wrapper kết nối Neo4j/FalkorDB, quản lý connection pool, retry logic. Toàn bộ code khác import từ đây thay vì tự mở connection.
search.py — implement Hybrid Search Pipeline 3 bước: Dual-Domain Search (query song song trên cả FAQ index và Document index), Graph Expansion (duyệt PREV/NEXT chunk liền kề và MENTIONS entity), Late Reranking (gọi reranker model, filter ngưỡng ≥ 0.7). Đây là file quan trọng nhất của B1.
reranker.py — load BGE Reranker hoặc tương đương, expose hàm rerank(query, candidates) -> List[scored]. Tách riêng để dễ swap model sau này.
embedding.py — load embedding model (Qwen3-Embedding-0.6B hoặc tương đương), expose hàm embed(texts) -> vectors. Có thể serve qua TEI hoặc tự build — đặt config trong config.yaml.

src/ingestion/
Code của module B1.3 — Data Ingestion Pipeline. Tách hoàn toàn khỏi src/graph/ vì đây là tác vụ offline, chạy một lần (hoặc khi có data mới), không phải serving path.
menu_loader.py — đọc menu CSV/JSON, tạo embedding cho từng MenuItem, ghi vào Graph DB với đầy đủ properties (tên, giá, size, category, ingredients).
faq_loader.py — đọc FAQ CSV, tách cặp Q&A, tạo embedding cho cả câu hỏi lẫn câu trả lời, lưu vào Chunk nodes kèm metadata (source, domain=faq).
doc_loader.py — xử lý PDF/DOCX: Semantic Chunking theo gradient breakpoint, gọi LLM để extract Entity tự động, tạo NEXT relationships giữa chunk liền kề, tạo MENTIONS relationships giữa chunk và entity.
deduplicator.py — Entity Deduplication bằng Jaccard similarity ≥ 0.85. Trước khi insert entity mới thì check xem đã tồn tại entity tương tự chưa, nếu có thì merge thay vì tạo node mới.
watcher.py — Watch Mode: dùng watchdog hoặc asyncio polling để detect file mới trong thư mục data/raw/, tự động trigger đúng loader tương ứng theo extension (.csv → menu/faq loader, .pdf/.docx → doc_loader).

src/serving/
Code của module B2.1 và B2.2 — LLM Serving Infrastructure.
sglang_client.py — HTTP client gọi SGLang server (hoặc vLLM). Không gọi model trực tiếp mà luôn qua đây — đảm bảo retry, timeout, error handling nhất quán. Expose hai hàm chính: complete(prompt) cho non-streaming và stream(prompt) cho SSE.
stream_handler.py — xử lý SSE stream token-by-token đúng spec: parse data: prefix, bắt và bỏ qua [DONE] signal (không parse như JSON), buffer và yield từng token ra ngoài. Đây là nơi duy nhất trong codebase xử lý raw SSE.
clause_splitter.py — phục vụ TTS downstream: nhận token stream, detect dấu câu (. , ? ! ;), yield từng clause hoàn chỉnh thay vì từng token. Order Agent và Consultant Agent gọi qua đây nếu có TTS.
vram_manager.py — tracking VRAM usage theo thời gian thực, expose metric để health check endpoint đọc. Không tự giải phóng VRAM (đó là việc của SGLang/vLLM) nhưng cung cấp signal để request queue điều tiết.

src/cache/
Code của module B2.3 và toàn bộ C2.2 — Multi-layer Caching System.
semantic_cache.py — Layer 4 của multi-layer cache, đây cũng là Semantic Cache cho B2.3 và C2.2. Lưu trữ embedding của các câu hỏi đã được trả lời. Khi có query mới thì embed query, tính cosine similarity với tất cả cached entries, nếu ≥ 0.95 (B2.3) hoặc ≥ 0.92 trên phần {Hành động} (C2.2) thì trả cache hit. Dùng FAISS hoặc hnswlib để tìm kiếm vector nhanh trong RAM.
session_cache.py — Layer 4 (in-memory) cho session history. Thực ra đây là bridge sang src/session/store.py — đặt ở đây để nhóm tất cả caching concern vào một chỗ, tránh logic cache bị rải rác khắp codebase.
invalidator.py — Cache Invalidation logic: khi menu hoặc FAQ thay đổi (watcher.py phát signal), invalidator xóa tất cả semantic cache entries liên quan đến domain đó. Implement theo tag-based invalidation — mỗi cache entry được tag theo domain (menu/faq/doc) để xóa theo nhóm.
stats.py — tracking cache hit rate, miss rate, latency của cache-hit path vs cache-miss path. Expose metrics cho health check và benchmark report.

src/intent_extraction/
Code của module C2.1 — SLM Fine-tuned cho Intent Extraction. Tách hoàn toàn khỏi src/router/ vì đây là model khác, tác vụ khác (structured extraction, không phải classification).
model.py — load SLM < 3B đã fine-tune cho intent extraction, quản lý vòng đời model.
extractor.py — nhận câu đầu vào, gọi model, parse output thành 3 thành phần cấu trúc: {subject}, {action}, {context}. Output là dict chuẩn. Phần {action} sẽ được semantic_cache.py dùng làm cache key, phần {context} được gửi kèm sang agent.
prompts.py — system prompt và few-shot examples cho tác vụ extraction. Ví dụ minh họa từ đề bài đặt ở đây để dễ tune.
paraphraser.py — Cache Hit path của C2.2: nhận response template từ cache + phần {context} từ extractor, kết hợp và sinh câu trả lời hoàn chỉnh dạng paraphrase tự nhiên, thân thiện, phù hợp F&B, phù hợp phát qua TTS. Xử lý hoàn toàn trong RAM, không gọi LLM Generator.

src/guardrails/
Code của module C3 — Production Guardrails.
tts_preprocessor.py — normalize text trước khi gửi TTS: "49.000đ" → "49k", "M" → "mờ", "VAT" → "thuế", xử lý các ký tự đặc biệt khác không phát âm được.
rate_limiter.py — rate limiting cho FastAPI endpoints, dùng slowapi hoặc implement thủ công bằng Redis counter. Config số request/phút per endpoint trong config.yaml.
fallback.py — Graceful Degradation: nếu Generator quá tải (queue full hoặc timeout), Router fallback về response template cứng dựa trên intent đã classify được. Ít nhất user nhận được câu trả lời tạm thay vì lỗi 500.
health.py — health check endpoints /health/router, /health/generator, /health/graph, /health/cache. Mỗi endpoint check đúng service của nó, trả HTTP 200/503 để load balancer hoặc Docker healthcheck đọc.

training/ (mở rộng)
sft_intent.py — fine-tune SLM < 3B cho tác vụ Intent Extraction (C2.1), tách riêng khỏi sft_train.py vốn dành cho Router. Cùng infra H100 nhưng data format và loss khác nhau.
evaluate_intent.py — đánh giá SLM Intent Extraction: accuracy trên 3 thành phần (subject/action/context accuracy riêng lẻ và combined), test trên bộ F&B thực tế.

data/ (mở rộng)
graph/ — dump/export từ Neo4j dùng cho backup và testing offline. Không import vào serving, chỉ dùng cho dev và CI.
intent/ — tương tự data/synthetic/ nhưng cho tác vụ C2.1: các cặp (input câu, output 3 thành phần). Sinh bằng pipeline tương tự A1.2 nhưng label format khác.
cache_test_set/ — 500 câu hỏi F&B thực tế có biến thể ngữ cảnh, dùng để đo cache hit rate ≥ 60% như tiêu chí C2.
